# EzBoost Improvement Plan

## Summary
EzBoost is now in a different state than the original improvement roadmap assumed. The core trust and UX work is substantially advanced:

- account flow is safer than before, with hashed-password support and better profile validation
- room data is user-scoped and imports are previewed before commit
- the import flow is guided and readiness-aware
- multiplier, segment, event, import, and BoostMe screens have had a cohesive frontend refresh
- optimization output now exposes more of the demand, bounds, and event-adjustment story

The accepted product scope for the current thesis build is:

- **seasonal GA optimization**
- **monthly forecast by historical month mapping**
- **event-based season overrides**

EzBoost is **not** attempting month-by-month re-optimization at this stage.

## Current Product Position
### What is now true
- Monthly history import is used for:
  - season classification
  - demand-curve fitting when enough valid data exists
- Room inventory import is used for:
  - sellable capacity
  - base ADR, min ADR, and max ADR constraints
  - hotel-specific multiplier previews
- The pricing engine remains GA-based:
  - it searches seasonal prices per room type
  - it respects imported bounds
  - it can use a fitted demand curve or the fallback default curve
- The monthly forecast is generated by:
  - mapping calendar months to historically learned seasons
  - applying event overrides that only bump seasons upward
  - using the optimized seasonal prices for the resulting month-season assignment

### What remains intentionally out of scope for now
- month-by-month price re-optimization
- room-specific optimizer overrides in the UI
- segment-specific optimizer search
- broader forecasting redesign beyond the current seasonal GA model

## Remaining Roadmap
### Block 1: BoostMe Cleanup
Objective:
- reduce view-layer risk without changing optimization behavior

Deliverables:
- move display-prep logic out of `BoostMe.jsp` and into servlet-prepared state
- keep the refreshed result UI intact while reducing scriptlet-heavy rendering
- make result rendering easier to reason about and easier to test

Acceptance criteria:
- BoostMe renders from prepared request attributes rather than mixed display logic
- no visible regression in result, forecast, or export behavior
- seasonal GA with monthly mapping remains unchanged

### Block 2: Flow Hardening
Objective:
- add stronger regression protection around the user journey the supervisor will see

Deliverables:
- add integration-style coverage for:
  - import preview -> commit
  - optimize -> result render
  - export after optimization
  - event override impact on monthly forecast
  - user isolation across imported data and results

Acceptance criteria:
- the highest-risk flows have automated regression coverage
- test coverage reflects the current user journey instead of only isolated utility logic

### Block 3: Final Thesis and Plan Alignment
Objective:
- keep documentation, product behavior, and thesis explanation aligned

Deliverables:
- update technical and project documents to consistently describe:
  - seasonal GA optimization
  - monthly forecast by mapping plus event overrides
  - fallback demand-curve behavior
  - current limitations and remaining cleanup work
- remove outdated plan statements that imply large unresolved UX problems already addressed in the product

Acceptance criteria:
- project documents match the current codebase
- the supervisor explanation and written roadmap tell the same story

## Recommended Execution Order
1. BoostMe cleanup
2. Flow hardening
3. Final thesis and plan alignment polish

## Success Criteria
The remaining work should be considered successful when:

1. the supervisor demo can be explained without opening source code
2. BoostMe is easier to maintain without changing the algorithm
3. the import -> optimize -> forecast -> export flow has meaningful regression protection
4. the roadmap no longer describes already-completed UX rescue work as if it were still pending

## Notes For The Current Build
- Seasonal GA with monthly mapping is the approved implementation unless the supervisor explicitly asks for a deeper forecasting model.
- Future work should prioritize demo safety, maintainability, and correctness proof over new feature expansion.
